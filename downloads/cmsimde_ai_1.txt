## 🧭 一、協同編輯 Git 流程總覽

六名組員可以透過 **Git** 進行以下兩種開發方式：

1. **本地端開發：** 將整個 repo clone 到本機，啟動 Flask 編輯器修改內容。
2. **GitHub Codespaces：** 在雲端環境進行開發，省去本地環境設定。

不論哪種方式，都會透過 Git 進行版本控制與同步，因此 Git 的使用流程需一致，以下為推薦的協作模式：

---

## 🧱 二、Git 分支策略建議

採用簡單的 **Feature Branch 工作流程**（適合內容協作）：

* `main`：穩定分支，儲存生成靜態頁面用的內容。
* `content-update/xxx`：每位組員針對特定內容頁面開新分支，如 `content-update/homepage-title`。

---

## 🛠 三、常用 Git 指令與使用情境

### 🔁 1. 初始化與取得最新專案內容

每位組員首次作業或準備修改前應執行：

```bash
# Clone 專案（第一次）
git clone https://github.com/your-org/your-repo.git
cd your-repo

# 確保 main 為最新
git checkout main
git pull origin main
```

### 🌿 2. 建立功能分支（內容分頁維護）

針對內容修改前，建立分支：

```bash
# 建立並切換到新分支
git checkout -b content-update/section-01
```

### 🧑‍💻 3. 開發與內容編輯

* 在本機或 GitHub Codespaces 中開啟 Flask 編輯器。
* 完成內容修改與預覽，確認無誤。

> Flask 編輯器的修改會影響 `config/content.htm`，以及自動生成的 `content/` 目錄靜態頁面。

### ✅ 4. 新增變更並提交

```bash
# 查看變更
git status

# 新增要追蹤的檔案（視情況選擇 config/ 與 content/）
git add config/content.htm content/

# 提交訊息說明修改內容
git commit -m "Update section 01 title and split into pages"
```

### 🔄 5. 與遠端同步並發起 PR

```bash
# 推送分支到 GitHub
git push origin content-update/section-01
```

接著至 GitHub 發起 **Pull Request（PR）**，由其他組員進行 review。

---

## 🧹 四、合併前同步主分支內容以避免衝突

若 PR 尚未合併，而 main 分支已有其他人合併的更新，請先同步：

```bash
# 切換回 main 並拉最新
git checkout main
git pull origin main

# 回到自己的分支
git checkout content-update/section-01

# 合併 main 的變更進來
git merge main

# 解決可能的衝突，完成後再提交
git commit -am "Resolve merge conflicts"
```

---

## 📤 五、合併至 main（由維護者或原作者完成）

經審查通過後可在 GitHub 上合併 PR 至 `main` 分支。合併後，GitHub Action 或本機可觸發部署流程（如有 CI/CD 設定）。

---

## 🔃 六、其他 Git 協作技巧

### 👯‍♀️ 同步他人修改（在開發中）

```bash
# 保持 main 最新
git checkout main
git pull origin main

# 回到開發分支
git checkout content-update/section-01

# 合併最新 main
git merge main
```

### 🧽 清理本地已完成的分支

```bash
# 本地刪除已合併的分支
git branch -d content-update/section-01

# 遠端刪除（如有權限）
git push origin --delete content-update/section-01
```

---

## 🤝 七、組員協同作業注意事項

1. **每次作業前一定要拉最新的 `main` 分支！**
2. **每次修改都要建立新的 feature 分支，避免直接動 `main`。**
3. **內容變動的區塊避免多人同時修改，降低衝突風險。**
4. **Pull Request 前需自測靜態內容是否正確產生。**
5. **Git Commit 訊息清楚表述修改內容（如：更新哪一段落標題與分頁）。**
6. **每人負責特定區域（如首頁、章節一、章節二），並記錄負責範圍。**
